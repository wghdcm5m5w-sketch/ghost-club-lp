# TetsuLog 第3次レビュー — 運用と整合性の死角

> 機能が揃った今こそ。「使い込むと壊れる」「集計が静かに嘘をつく」「言ってることとコードが違う」を暴く。
> 第1弾=機能不足、第2弾=懐疑派の不信、第3弾=**運用・整合性・誠実性**。

---

## 実際に見つけた「コードのバグ・嘘」

### ★1. ウィジェットの「累計」が遭遇件数を編成数のように見せている（実バグ）
- `CollectionWidget.loadEntry()` の `totalFormations` は `fetchCount(FetchDescriptor<Sighting>())` ＝ **遭遇記録の総数**。
- 同じ編成を10回撮れば10カウント。これを「編成」と誤認させる変数名・文脈。
- **修正**: 遭遇済みのユニーク編成数を数える。ラベルも明確化。

### ★2. 「データは手元に残る」と言うが、写真はエクスポートに含まれない（嘘）
- `ExportService` は `photoFilenames`（ファイル名）しか書き出さない。画像本体は端末Documents。
- 端末を変えたら**写真は移行できず、エクスポートにも入っていない**。LP/設定の「データは手元に残る」と齟齬。
- **修正**: 設定文言を正直に（「写真は別途バックアップを」）。将来は画像同梱エクスポートを検討と明記。

### ★3. 保存エラーを `try? context.save()` で握りつぶしている（静かなデータ消失）
- 全箇所 `try?`。CloudKit競合や容量不足で保存に失敗しても**ユーザーに伝わらず、記録が消える**。
- **修正**: 主要な保存を do/catch し、失敗時にアラートとエラーハプティクス。

## 使い込むと露呈する設計問題

### A. 駅名・路線名のフリーテキスト表記ゆれ（第1弾で指摘・未対応）
- 「新宿」「JR新宿」「新宿駅」が別物として集計され、統計の「よく行く駅」が割れる。
- **対処（今回）**: 過去入力からのサジェストを出し、表記を揃えやすくする。

### B. `@Query` 全件ロード × 線形フィルタ
- 数千件で図鑑/記録/統計が全件メモリ展開＋線形走査。スクロールとフィルタが重くなる。
- **対処**: 今回は許容（個人の記録は数千〜数万が上限、SwiftDataは耐える）。将来 `FetchDescriptor` の predicate/limit を導入。レビューに明記。

### C. 同一編成の重複記録に無警告
- 同じ日に同じ編成を二度記録しても何も言わない。意図的な複数記録もあるので**エラーにはしない**が、直近の同編成記録を気づかせる余地。
- **対処**: 今回は見送り（誤検知リスク）。将来オプション化。

### D. 音鉄（録音）対応がゼロ
- VVVF音・走行音・車内放送を残したい層が完全に置き去り。
- **対処**: 今回はモデルに録音参照の余地を残すに留め、v2.0で録音添付を正式対応とレビューに記載。メモ欄で暫定対応可能。

### E. CLMonitor/ジオフェンス20件上限
- 撮影地が20を超えると監視されない。
- **対処**: 今回は撮影地が少ない前提で許容。将来「現在地から近い順に20件」へ。レビューに明記。

- 月別チャートのバーが VoiceOver で読めない（数値ラベルなし）。→ `accessibilityLabel` 追加。

## ビジネス・運用の死角（記録のみ・実装外）

- 980円買い切りで継続開発の原資が薄い。サポート問い合わせの一次対応コスト。
- 自前JSON以外（レイルラボ/乗りつぶしオンラインのCSV）からの移行は未対応 → 乗り換え障壁。
- 撮影地共有の倫理（私有地・人物写り込み）。現状は個人用のみなので低リスクだが、共有機能を足すなら要設計。
